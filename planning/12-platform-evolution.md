# رؤية منصة Gateo وخطة تطويرها المستقبلية

> تمت إضافة هذا الملف بناءً على الملف المرفق `تخطيط جديد.odt`، وتم دمجه مع البنية البرمجية الحالية للمشروع بحيث تبقى التوسعات المستقبلية آمنة ومتسلسلة.

---

## 1. الرؤية العامة للمنصة

**Gateo** هي منصة اجتماعية احترافية متعددة القطاعات، تهدف إلى ربط الأفراد والمهنيين والأنشطة التجارية ضمن بيئة موحدة قابلة للتوسع.

المنصة ليست موجهة لقطاع التجميل فقط، بل تم تصميمها لاستيعاب أي قطاع حالي أو مستقبلي مثل:

- التجميل والعناية الشخصية.
- القطاع الطبي والصحي.
- القطاع العلمي والبحثي.
- التعليم والتدريب.
- الهندسة والتقنية.
- الخدمات المهنية والحرفية.
- التجارة الإلكترونية.
- أي قطاع جديد يتم إضافته مستقبلاً.

تم بناء الهيكل البرمجي للمنصة بطريقة مرنة تسمح بإضافة أقسام وفئات وخدمات جديدة دون الحاجة إلى إعادة بناء النظام.

## 2. أنواع الحسابات داخل المنصة

النظام يدعم حالياً ثلاثة أنواع رئيسية من الحسابات، وهي مرتبطة مباشرة بالبنية البرمجية الحالية:

### 2.1 الحساب الشخصي

- الحساب الأساسي لجميع المستخدمين.
- يتيح إنشاء ملف شخصي، التواصل الاجتماعي، إضافة الأصدقاء، التفاعل مع المحتوى، ومتابعة الاهتمامات.
- لا يظهر في دليل الأعمال ولا يمكنه إنشاء موقع تجاري.
- **الارتباط التقني:** `User.accountType = USER` + جدول `Profile`.

### 2.2 الحساب الاحترافي (دليل الأعمال)

- حساب مخصص للأفراد المهنيين وأصحاب الخبرات الذين لا يمتلكون بالضرورة نشاطاً تجارياً مسجلاً.
- أمثلة: محامون، أطباء، مهندسون، معلمون، مصممون، مبرمجون، سباكون، خبراء تجميل، مترجمون، باحثون أكاديميون.
- يعرض خبرات وخدمات وشهادات ومؤهلات المهني.
- **الارتباط التقني:** `User.accountType = PROFESSIONAL` + جدول `ProfessionalProfile`.

### 2.3 الحساب التجاري

- مخصص للأنشطة التجارية والشركات والمؤسسات التي تمتلك كياناً فعلياً.
- يمكنه إنشاء موقع إلكتروني احترافي أو متجر إلكتروني داخل المنصة.
- أمثلة: صالونات تجميل، عيادات، متاجر، مطاعم، أكاديميات، مصانع، شركات تقنية.
- **الارتباط التقني:** `User.accountType = BUSINESS` + جدول `Business`.

| الميزة | دليل الأعمال (احترافي) | النشاط التجاري |
|---|---|---|
| الكيان | فرد | شركة/نشاط |
| العرض | خبرات وخدمات | نشاط تجاري كامل |
| يحتاج سجل تجاري | لا | نعم |
| الصفحة | صفحة احترافية | موقع/متجر متكامل |
| التركيز | الشخص | المؤسسة |

## 3. رحلة المستخدم داخل المنصة

### المرحلة الأولى: التسجيل والدخول

1. التسجيل.
2. التحقق من الحساب.
3. الدخول للمنصة.
4. إنشاء الملف الشخصي.

### المرحلة الثانية: استكمال الملف الشخصي

- الصورة الشخصية، النبذة، الموقع، الاهتمامات، التخصصات.
- بعد الإكمال: تفعيل الحساب، اقتراحات الأصدقاء، وبدء تجربة التواصل الاجتماعي.

### المرحلة الثالثة: شاشة تطوير الحساب

بعد استكمال الملف الشخصي، يعرض النظام خيارات تطوير الحساب:

| الخيار | الوصف |
|---|---|
| **الحساب الشخصي** | الاستمرار في استخدام المنصة الاجتماعية فقط. |
| **الانضمام إلى دليل الأعمال** | عرض الخبرات والخدمات المهنية بعد الموافقة على الشروط وإرسال الطلب للمراجعة. |
| **تسجيل نشاط تجاري** | إنشاء موقع إلكتروني أو متجر احترافي بعد رفع الوثائق واعتماد النشاط. |

> ملاحظة تقنية: لا يتم تغيير جدول `User` الأساسي عند التطوير؛ بل يتم إنشاء سجل جديد في `ProfessionalProfile` أو `Business` مع تحديث `accountType` فقط.

## 4. نظام إنشاء المواقع التجارية

### الخطوة الأولى: اختيار المجال

أمثلة: صالون تجميل، عيادة، مطعم، متجر، شركة تقنية، مركز تدريب، مكتب هندسي.

### الخطوة الثانية: تحليل احتياجات العميل

يقوم النظام بجمع:

- نوع النشاط.
- الخدمات.
- المنتجات.
- الجمهور المستهدف.
- الهوية البصرية.

### الخطوة الثالثة: توليد قالب ذكي مخصص

بدلاً من اختيار قالب ثابت فقط، يقترح النظام:

- تصميم مناسب.
- صفحات مناسبة.
- أقسام مناسبة.
- ألوان مناسبة.

بحيث يحصل كل نشاط على موقع يناسب تخصصه.

أمثلة:

- `gateo.site/jojo`

### الخطوة الخامسة: لوحة التحكم الاحترافية

تشمل:

- إدارة الصفحات.
- إدارة الخدمات.
- إدارة المنتجات.
- الحجوزات.
- الرسائل.
- العملاء.
- الإحصائيات.
- العروض.
- المحتوى.

## 5. متجر Gateo المركزي (مستقبلي)

مستقبلاً يتم إنشاء سوق مركزي للمنصة.

يقوم النظام تلقائياً بـ:

- قراءة المنتجات المعروضة بالمواقع.
- تصنيفها حسب المجال.
- عرضها داخل متجر Gateo.

وعند الضغط على المنتج:

- يتم تحويل المستخدم إلى الموقع الأصلي.
- تتم عملية الشراء أو الحجز مباشرة من موقع صاحب النشاط.

وبذلك يحتفظ صاحب النشاط باستقلاليته الكاملة.

---

## 6. متطلبات البناء البرمجي

يجب تصميم النظام على أسس تجعل التوسع مستقبلاً آمناً:

### 6.1 Modular Architecture

كل قسم عبارة عن وحدة مستقلة:

- التواصل الاجتماعي.
- دليل الأعمال.
- المواقع الإلكترونية.
- المتاجر.
- الحجوزات.
- الأبحاث العلمية.
- الذكاء الاصطناعي.
- الإعلانات.

بحيث يمكن إضافة وحدات جديدة مستقبلاً دون التأثير على النظام الأساسي.

### 6.2 Dynamic Categories

تصنيفات ديناميكية قابلة للإضافة والحذف والتعديل دون تدخل برمجي.

**الارتباط التقني الحالي:** جداول `Category` و `Subcategory` مع دعم `isLadiesGate` و `sortOrder`.

### 6.3 Dynamic Fields

حقول متغيرة حسب التخصص والمجال.

**الاستراتيجية المقترحة:**

- الاستمرار في استخدام حقول `Json` للحقول الاختيارية (مثل `specializations`، `services`، `workExperience`).
- عند الحاجة إلى حقول معقدة ومفهرسة، يتم إنشاء جدول `ProfileField` / `BusinessField` منفصل (نموذج EAV) بدلاً من تعديل الجداول الأساسية.
- هذا يحافظ على استقرار `schema.prisma` الأساسي.

### 6.4 Dynamic Themes

قوالب ذكية قابلة للتخصيص والتطوير المستمر.

**الاستراتيجية المقترحة:**

- تخزين إعدادات المظهر في جدول `BusinessTheme` (مستقبلي) مرتبط بـ `Business`.
- الاعتماد على Tailwind CSS + CSS variables لتطبيق الألوان والخطوط ديناميكياً.
- عدم تعديل المكونات الأساسية؛ بل تمرير إعدادات المظهر كـ `props` أو `context`.

## 8. مبادئ التطوير المحافظة على البنية

1. **عدم تعديل الجداول الأساسية:** عند إضافة ميزة جديدة، يتم إضافة جداول جانبية أو استخدام حقول `Json` بدلاً من تغيير هيكل `User` أو `Business` أو `Profile`.
2. **فصل الميزات الجديدة:** كل قطاع أو وحدة جديدة في مجلد مستقل تحت `src/features/` أو `src/app/(features)/`.
3. **إعادة الاستخدام:** استخدام الـ APIs والمكونات المشتركة (مثل البحث، المراسلة، الحجوزات) بدلاً من إعادة بنائها.
4. **التجريد:** الاعتماد على `accountType` و `role` للتحكم في الصلاحيات بدلاً من إنشاء منطق مخصص لكل نوع حساب.
5. **التراجع الآمن:** كل تغيير يكون قابلاً للتراجع عبر migrations منفصلة واختبارات تلقائية.
6. **عدم كسر العقود العامة:** الحفاظ على شكل الـ API responses والـ URLs العامة للمواقع التجارية.
